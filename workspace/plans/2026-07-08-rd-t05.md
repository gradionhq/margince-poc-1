# RD-T05 — Backend — attachments: blob-seam references, presigned upload/download, scan gate, provenance, download-audited (RD-WIRE-1) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use
> checkbox (`- [ ]`) syntax for tracking. **Commit steps are authored unticked with no SHA — the
> swarm tooling derives the ledger from `Ledger-Task: N` git trailers. Never hand-tick a checkbox
> or hand-type a SHA.**

**Goal:** Implement the handler + business logic behind the already-minted (RD-T02) `/attachments`
contract surface (`listAttachments`, `createAttachment`, `getAttachment`, `archiveAttachment`) in
`backend/api/crm.yaml`. The `attachment` table already exists (migration `000009`); this plan adds
the missing `scan_status` DB column, extends `blobstore.Store` with presigned-URL methods, builds a
brand-new `backend/internal/modules/records` module (domain/ports/adapters/transport — currently only
test-only files from RD-T08), wires it into routes/RBAC/seed exactly like the just-merged `offers`
module (OP-T04), and reuses the `activities` module for the download-audit-as-timeline-activity
requirement. **No `crm.yaml` change.**

**Scope:** be-only. Every task touches only `backend/**` + one new `workspace/manual-test/rd-t05.md`
guide — no frontend. Per-task type/build gate = `go build ./backend/...` (workspace-wide from Task 6
on, since that task edits `all_operations.go`/`routes.go`/`server_impl.go`/`main.go`, shared
cross-package files). FINAL GATE keeps `make check-q`; FINAL-UAT coverage keeps; live-stack UAT runs
the full stack (`make test-integration` must stay 0-skip, real MinIO); craftsmanship review is the
full agent; fe-uat capture lane is skipped (no FE surface).

**Architecture:** New module `backend/internal/modules/records/{domain,ports,adapters,transport}` +
`module.go`, D6 shape mirroring `offers`/`activities`. `records` depends on `activities` (to call
`activities.NewActivityStore(db).Create` for the download-audit) and on `platform/blobstore` (to
mint presigned URLs) — both new, additive arch-lint DAG edges; no module depends on `records` in
return, so the DAG stays acyclic. Archive-cascade (DM-CONV-15) is implemented as raw literal SQL
`UPDATE attachment ...` statements added to the five *existing* owning modules' `Archive` methods
(`people`, `organizations`, `deals`, `leads`, `activities`), not a new dependency edge — this mirrors
the only existing cascade precedent (`PersonStore.Archive` → `person_email`/`person_phone`) and keeps
`records` a leaf the other five modules never need to import.

**Tech Stack:** Go 1.x, `database/sql` + `lib/pq`, Postgres, MinIO (minio-go v7.2.1, already a
dependency). Store tests are `//go:build integration` (`pgtest.OpenTestDB`), run via
`make test-it DIR=backend/internal/modules/records` or the full `make test-integration`. Handler
tests are plain (no build tag), in-memory-store-seam style mirroring
`activities/transport/handler_activity_test.go`.

## Global Constraints — read before starting any task

These are the concrete design decisions this plan makes to resolve gaps the spec explicitly left
open ("the handler ticket decides"), confirmed against the actual codebase state on this branch
(`swarm/rd-t05`, forked from a clean `main`, no commits ahead as of this plan). **If any step reveals
a genuine need to redesign the DB schema/contract beyond what's decided here, STOP and escalate — do
not force it through.**

1. **`scan_status` lands as an on-row column** (per the contract's own "either is acceptable" note),
   via new migration `000076_attachment_scan_status.up.sql` (confirmed next-free number: `ls
   backend/migrations | tail` shows `000075_formula_field_boundary` is the newest; **no**
   `000009_lists_tags_attachments.up.sql` edit). `scan_status text NOT NULL DEFAULT 'scanning' CHECK
   (scan_status IN ('scanning','clean','blocked'))`.

2. **Blob bucket: reuse the existing `transcripts` bucket via a `attachments/` key prefix — no
   infra/docker-compose change.** `infra/docker-compose.dev.yml`'s `minio-init` only auto-provisions
   a bucket named `transcripts` (`BLOBSTORE_BUCKET` env matches). `blobstore.Store.Put`/`Get`/the new
   presign methods take an opaque `key` — the records module composes `storage_key` as
   `attachments/<workspace_id>/<attachment_id>/<filename>`, giving every attachment object its own
   key inside the one bucket already wired end-to-end. This avoids provisioning a second bucket for
   a ticket explicitly scoped to backend Go code.

3. **`blobstore.Store` presign extension** (Task 2, independent of the rest): add two methods to the
   `Store` interface —
   `PresignedPutURL(ctx context.Context, key string, expiry time.Duration) (string, error)` and
   `PresignedGetURL(ctx context.Context, key string, expiry time.Duration) (string, error)`.
   `MinIOStore` implements both via `s.client.PresignedPutObject(ctx, s.bucket, key, expiry)` /
   `s.client.PresignedGetObject(ctx, s.bucket, key, expiry, url.Values{})` (minio-go v7's real
   presign API — `github.com/minio/minio-go/v7` is already imported in `minio.go`), returning
   `u.String()`. `MemoryStore` returns a synthetic, deterministic `memory://<key>?expires=<unix>`
   string (no real HTTP semantics — it's a dev/test fake; document this in a doc comment) so
   `blobstore_test.go`'s existing round-trip tests stay meaningful and new presign tests can assert
   on the URL shape without a live server.

4. **Virus-scanner seam lives in `records/ports`, not `platform/blobstore`** (scanning is a
   records-depth concern per the spec). `ports.Scanner interface { Scan(ctx context.Context,
   storageKey string) (status string, err error) }` — `status` must be `"clean"` or `"blocked"`
   (never `"scanning"`, that's the row's own default). No real scan happens on `createAttachment`
   (there is no scanner to call synchronously, and bytes aren't even uploaded yet at create time —
   two-phase upload). The seam is exercised via a new **internal, non-HTTP** store method
   `AttachmentStore.MarkScanResult(ctx, id, workspaceID string, scanner ports.Scanner) (domain.Attachment,
   error)` that calls `scanner.Scan(ctx, row.StorageKey)`, validates the returned status, and persists
   it — this is what integration tests call directly to simulate a scan verdict (no new public HTTP
   endpoint, matching the spec's explicit instruction). `adapters.FakeScanner` (a small
   map-keyed-by-storage-key or fixed-result fake) ships for dev/tests; there is no dev-mode
   auto-`clean` path anywhere — a freshly created row stays `scanning` until a test/administrator
   calls `MarkScanResult`.

5. **Download-audit reuses `activities.NewActivityStore(db).Create` directly** (the concrete
   reusable entry point — confirmed `cmd/api/routes.go` constructs `activities.NewActivityStore(k.db)`
   inline everywhere, not via `activities/app.Service`, which has no `Create` method at all). Kind
   `"note"` (the closest of the seven real DB-enum values — `email|call|meeting|note|task|whatsapp|
   telegram`; there is no `"system"` kind, confirmed against `migrations/000003_core_objects.up.sql`'s
   CHECK and `activities/domain/domain.go`'s own comment). Subject: `"Attachment downloaded: <filename>"`.
   Provenance: `source: "system"`, `captured_by: "system:attachment-download-audit"`.

   **Hard schema limitation, confirmed and accepted (do not attempt to work around it with a new
   migration — out of scope):** `activity_link`'s own DB CHECK (`activity_link_shape`,
   `migrations/000003_core_objects.up.sql`) only allows `entity_type IN ('person','organization',
   'deal')`, each with exactly one non-null FK column. `attachment.entity_type` additionally allows
   `lead`/`activity` (DM-CONV-17's full 5-value canonical vocabulary). Therefore:
   - Attachment bound to **person/organization/deal**: the download-audit `Activity` is created
     **with** a `Links: []domain.ActivityLink{{EntityType: entityType, EntityID: entityID}}` entry —
     it lands in `activity_link` and appears on that record's 360 timeline via the existing
     `ActivityStore.List(ctx, workspaceID, entityType, entityID, ...)` read path — **no code change
     needed there**, this "just works" once the link is written (RD-AC-9 satisfied for these three
     types).
   - Attachment bound to **lead/activity**: the download-audit `Activity` is created **with an empty
     `Links` slice** — the row exists (audited: `GET /activities/{id}` can fetch it directly) but does
     **not** appear on any timeline, because **no timeline-read mechanism exists for `lead`/`activity`
     entity types anywhere in this codebase today** (`ActivityStore.List` errors
     `"unknown entity_type"` for anything outside person/organization/deal — a pre-existing platform
     limitation, not something this ticket introduces or can fix without widening
     `activity_link_shape`, which is out of scope). **This is a known, accepted, documented gap** —
     call it out again in the report, exactly like OP-T04's `offer_template` provenance gap. The
     integration test for RD-AC-2/RD-AC-9's "appears in timeline" bullet is scoped to a
     person/organization/deal-bound attachment (the case the platform actually supports); a
     lead/activity-bound download is tested only for "an Activity row was written" (audited), not
     timeline appearance.

6. **Visibility inheritance ("disclosed locked row") design — genuinely new logic, kept minimal:**
   - No existing "restricted/disclosed" runtime pattern exists anywhere in this codebase (confirmed
     by grep) — `RbacMiddleware`/`session.AuthorizePerms` (`shared/ports/session/session.go:112-119`)
     is **object-level only** (does the role have `read` on `attachment` at all); it never consults
     `RowScope` or `record_grant`. No single-record `GET` handler in the codebase today applies any
     row-level check either (confirmed: `OrgHandler.get` calls `store.GetAny` with zero visibility
     filtering) — so there is no existing "restricted record GET" behavior to match; this plan
     designs the check fresh, scoped tightly to attachments only, per the spec's "keep it minimal"
     instruction.
   - **The check is delegated to the bound record's own object/row_scope — not `attachment`'s own
     row_scope** (this is the literal reading of "an attachment's visibility is exactly its bound
     record's visibility"). New helper `adapters.RecordVisible(ctx context.Context, db *sql.DB,
     workspaceID, entityType, entityID string, principal crmctx.Principal, perms
     session.RolePermissions) (bool, error)`:
     - Look up `perms[entityType]["read"]` (entityType ∈ `{person,organization,deal,lead,activity}` —
       these strings already equal their own RBAC object names, confirmed in `auth.go`'s `Obj*`
       consts / `session.knownObjects`). Entry absent or `RowScope` invalid → not visible (the caller
       can't even read the bound record's own object).
     - `RowScope == "all"` → visible.
     - `entityType == "activity"` (no `owner_id` column, and `record_grant.record_type` CHECK
       excludes `'activity'` — confirmed `migrations/000069_record_grant.up.sql`) → any row_scope
       (`own`/`team`/`all`) that passed the object-level check above is treated as visible; the
       `activity` table has no per-row ownership model anywhere else in this codebase either, so this
       is consistent with existing product reality, not a new gap this ticket invents.
     - `entityType` ∈ `{person,organization,deal,lead}` and `RowScope` ∈ `{own,team}` (`"team"` has no
       working implementation anywhere in this codebase — confirmed zero query-site usages of
       `team_membership` outside `identity`; treat `team` the same as `own`, matching existing
       platform reality): query `SELECT owner_id FROM <table> WHERE id=$1 AND workspace_id=$2` (table
       name from a small literal switch, exactly like `activities/adapters`'s `insertLink` switch — no
       dynamic SQL), then visible iff `owner_id = principal.UserID` **OR**
       `EXISTS (SELECT 1 FROM record_grant WHERE workspace_id=$ws AND record_type=$entityType AND
       record_id=$entityID AND subject_type='user' AND subject_id=$userID AND (expires_at IS NULL OR
       expires_at > now()))` — the exact widening predicate already used by
       `organizations/adapters/store_org_list.go:buildOrgListWhere`'s `owner_id` filter, reused
       verbatim as an EXISTS check instead of a list WHERE-clause fragment. `record_grant.record_type`
       CHECK only allows `deal/person/organization/lead` — `activity` never reaches this branch (see
       above), so no CHECK violation risk.
   - **RLS scoping is mandatory for these two reads, not optional (PLAN-review correction, opus
     tier, caught before any code was written).** `person`/`organization`/`deal`/`lead`/`record_grant`
     are all `FORCE ROW LEVEL SECURITY` tables (`migrations/000003_core_objects.up.sql`'s
     `%_tenant_isolation` policies; `000069_record_grant.up.sql`), and the app pool runs as the
     non-`BYPASSRLS` `margince_app` role (`000004_app_role.up.sql`). The `app.workspace_id`/
     `app.user_id` GUCs these policies key off are **transaction-local only**, set exclusively inside
     `database.WithWorkspaceTx`/`SetWorkspaceScope` (`platform/database/database.go`). A bare
     `db.QueryRowContext` against these tables on the shared pool (no surrounding `WithWorkspaceTx`)
     sees an **unset** `app.workspace_id`, so every policy predicate evaluates to `NULL` and every row
     is invisible — an un-scoped `RecordVisible` would silently return `false` for every `own`/`team`
     case and every `record_grant` share, breaking exactly the visibility this task exists to prove.
     **`RecordVisible`'s implementation MUST wrap both reads in one
     `database.WithWorkspaceTx(ctx, db, workspaceID, func(tx *sql.Tx) error {...})` call and query via
     `tx`, never `db` directly** — this also sets `app.user_id` (needed by `record_grant`'s own
     widened RLS backstop) from `crmctx.From(ctx)`, so the caller's `ctx` must already carry the
     request's `crmctx.Principal` (it does — the handler's `ctx` comes straight from `r.Context()`
     after `RbacMiddleware`/session middleware ran). **Do not validate this with
     `pgtest.SetRLS`-only tests** — `SetRLS` sets `app.workspace_id` at the **session** level on the
     test `*sql.DB`, which would make a broken (session-unscoped) implementation pass its own test
     while still being broken in production (the pool never gets a session-level GUC there). Task 5's
     tests must include at least one case that does **not** call `pgtest.SetRLS` and instead relies
     solely on `RecordVisible`'s own internal `WithWorkspaceTx` scoping — proving the production code
     path, not just the test harness's more permissive session scoping.
   - **Wire shape of "disclosed, not omitted, not 404":** when `RecordVisible` returns false, the
     handler still returns the full `Attachment` row (id, entity_type/entity_id, filename,
     content_type, byte_size, storage_key, scan_status, source, captured_by, timestamps — all
     truthful, matching the contract's required-field list so no field goes non-conformant-null) but
     **`upload_url`/`download_url` are forced `null` regardless of `scan_status`**, and the
     download-audit write (Constraint 5) is **skipped** (no download occurred). This satisfies "shown
     as a disclosed row, not a 404, not silent omission" while never handing bytes access to a caller
     who can't see the bound record. `listAttachments` applies the same per-item rule (never a silent
     drop from the page). No "request-access path" is built — the contract has no such verb and
     `crm.yaml` is out of scope to extend; note this narrowing in the report.
   - This check runs on every `getAttachment`/`listAttachments` call (needs `perms` — call
     `platformauth.LoadRolePermissions(ctx, db, tenantID, userID)` once per request, same call the
     `RbacMiddleware` itself already makes one layer up — a second call here is the accepted cost of
     keeping the handler self-contained, matching no existing precedent for passing `perms` through
     context).

7. **P12 platform audit (`audit_log`) vs. RD-AC-2's "download-audited" are two separate mechanisms —
   both required, do not conflate them.** `createAttachment`/`archiveAttachment` are **mutations** on
   the `attachment` table and must each write exactly one `audit_log` row via
   `crmaudit.EntryFromPrincipal(ctx, "create"/"archive", "attachment", &id, nil, nil)` +
   `crmaudit.WriteTx` inside the same tx — mirroring `organizations`/`deals`/`offers` (majority
   precedent; `audit_log.action` CHECK already includes `create`/`archive`, no migration needed). This
   requires `recordsadapters: mayDependOn: [..., audit, ...]` in `.go-arch-lint.yml` (mirrors
   `offersadapters`, the newest precedent for this exact dependency). **The download-audit (Constraint
   5) is a completely different mechanism** (an `activity` row via the `activities` module, triggered
   on `get`/`list` reads, not on the mutation path) — do not write an `audit_log` row for downloads;
   the spec is explicit that this must reuse the activities module, not a bespoke audit mechanism.
   **No `event_outbox` writes** for `attachment.*` — no documented consumer (same reasoning OP-T04
   already established for `product`/`offer_template`; avoids inventing unscoped infrastructure).

8. **`Idempotency-Key` header declared on `createAttachment` — ignored, matching majority precedent**
   (only `deals` implements idempotency-key resolution; `people`/`organizations`/`relationships`/
   `activities`/`offers` all ignore it despite the contract declaring the param). Do not implement it
   here either.

9. **RBAC object wiring is FOUR-part for this codebase (the spec undercounts it as three) — get all
   four right up front, this is exactly the class of gap OP-T04 missed on its first pass (see
   `941b6b6 fix(identity): grant product/offer_template to bootstrap-admin permissions`):**

   2. `session.knownObjects["attachment"] = true` — `shared/ports/session/session.go:49-60`.
   3. **`backend/seed/dev.sql`'s FIVE seeded dev-workspace roles** (the spec's spec text says "four
      roles (bootstrap-admin, member, viewer, admin)" — that terminology does **not** match this
      repo's actual seed data; the five real seeded roles are `admin`, `rep`, `read_only`, `manager`,
      `ops`, lines ~100-129). Add an `"attachment"` entry to **`admin`, `rep`, `read_only`, `manager`**
      (mirror `product`/`offer_template`'s exact row_scope shape per role — see the Authorization

   4. **`backend/internal/modules/identity/transport/auth_handler.go`'s `adminPermissionsJSON`** — the

## Authorization Matrix — `attachment` object (RD-T05's access-touching surface)

`attachment` has exactly three real verbs: `read` (GET list/get), `create` (POST), `archive`
(DELETE) — no `update`. Deny = the object key is absent from that role's `permissions` JSONB (not an
explicit `false`) → `AuthorizePerms` returns `ErrForbidden` → 403, mirroring every other object.
Visibility beyond this object-level gate is governed entirely by Constraint 6's `RecordVisible` check
against the **bound record's own** object/row_scope, not `attachment`'s row_scope (documented there).

| Role | read (object-level gate) | create | archive |
|---|---|---|---|
| `admin` (dev.sql) | `row_scope: all` | `row_scope: all` | `row_scope: all` |
| `rep` (dev.sql) | `row_scope: own` | `row_scope: own` | deny (no entry) |
| `read_only` (dev.sql) | `row_scope: all` | deny | deny |
| `manager` (dev.sql) | `row_scope: all` | `row_scope: all` | deny (no entry) |
| `ops` (dev.sql) | deny | deny | deny |
| bootstrap-admin (`adminPermissionsJSON`, self-service `POST /workspaces`) | `row_scope: all` | `row_scope: all` | `row_scope: all` |

No `x-mcp-tool` tier is declared on any of the four attachment operations (confirmed: zero
`x-mcp-tool` hits in the `crm.yaml` block) — no MCP-surface reachability gate applies to this ticket.
`RbacMiddleware` applies identically regardless of `principal.IsAgent`; there is no new Agent Seat
Passport interaction beyond the existing global mechanism (ADR-0013) since no new tool tier is minted
here.

```go
// RecordVisible reports whether principal can see the bound record
// (entityType/entityID) per perms's row_scope for that object — Constraint
// 6: attachment visibility is exactly the bound record's visibility, not a
// separate ACL. entityType must be one of domain.EntityType{Person,
// Organization,Deal,Lead,Activity}.
func RecordVisible(ctx context.Context, db *sql.DB, workspaceID, entityType, entityID string, principal crmctx.Principal, perms session.RolePermissions) (bool, error) {
	entry, ok := perms[entityType]
	if !ok {
		return false, nil
	}
	rule, ok := entry.Actions["read"]
	if !ok {
		return false, nil
	}
	if rule.RowScope == "all" {
		return true, nil
	}
	if entityType == domain.EntityTypeActivity {
		return true, nil // no per-row ownership model for activity anywhere in this codebase
	}
	table, ok := map[string]string{
		domain.EntityTypePerson: "person", domain.EntityTypeOrganization: "organization",
		domain.EntityTypeDeal: "deal", domain.EntityTypeLead: "lead",
	}[entityType]
	if !ok {
		return false, nil
	}
	// RLS-scoping fix (PLAN review): person/organization/deal/lead/record_grant are all FORCE RLS,
	// keyed off app.workspace_id/app.user_id GUCs that only exist inside a WithWorkspaceTx-scoped
	// tx. A bare db.QueryRowContext here would see those GUCs unset and silently match zero rows.
	var visible bool
	err := database.WithWorkspaceTx(ctx, db, workspaceID, func(tx *sql.Tx) error {
		var ownerID sql.NullString
		err := tx.QueryRowContext(ctx, fmt.Sprintf(`SELECT owner_id FROM %s WHERE id=$1 AND workspace_id=$2`, table),
			entityID, workspaceID).Scan(&ownerID)
		if errors.Is(err, sql.ErrNoRows) {
			return nil // visible stays false: bound record not found/not visible under RLS
		}
		if err != nil {
			return err
		}
		if ownerID.Valid && ownerID.String == principal.UserID {
			visible = true
			return nil
		}
		return tx.QueryRowContext(ctx, `SELECT EXISTS (
			SELECT 1 FROM record_grant WHERE workspace_id=$1 AND record_type=$2 AND record_id=$3
			  AND subject_type='user' AND subject_id=$4 AND (expires_at IS NULL OR expires_at > now()))`,
			workspaceID, entityType, entityID, principal.UserID).Scan(&visible)
	})
	return visible, err
}
```
(`table` is a map keyed by a **fixed literal set**, never interpolating caller input beyond the
already-validated `entityType` enum — mirrors `activities/adapters`'s `insertLink` switch discipline;
double-check the fixed keys exactly match the five DB table names before committing. `ctx` passed to
`WithWorkspaceTx` must carry the request's `crmctx.Principal` — via `crmctx.With` — so
`SetWorkspaceScope` picks up `app.user_id`; the handler's `r.Context()` already does after
`RbacMiddleware`/session middleware ran.)
