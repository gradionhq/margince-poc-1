# T19 Manual UAT — Person 360

Prereqs: `make infra-up && make migrate-up && make seed-reset && make run` (backend), `make
fe-dev` (frontend), a seeded workspace with at least one person who has emails/phones, a
current-primary employment relationship, a non-null `strength`, at least one activity, and a
second person to use as a merge target. **Known gap:** `make seed-reset` alone does not seed
enough of the above (no orgs/employment-relationships/activities) — create them manually via
API+SQL to actually exercise this guide (see the Swarm Report's Escalations for the systemic
follow-up ticket).

## Known gaps (carried honestly, not worked around — see PR description)
- Company link (step 2) points to `/companies/{organization_id}`, which has no route on `main`
  yet.
- `mergePerson` (steps 9-14) has no `If-Match`/version param; its 409 uses the generic
  `Conflict` schema, so the dialog reads the actual `code` at runtime instead of assuming
  `version_skew`.
- `NotesTab` (step 8) has no confirmed create-note endpoint — Save is local-state only.
- `ActivityRef` (step 6) carries no `source`/`captured_by` — no per-row provenance chip.
- **`GET /people/{id}` never populates `emails`/`phones`** on the composite read (confirmed
  live via curl against a real DB with seeded rows) — the backend's `assembleComposite` /
  `PersonStore.Get`/`GetAny` never join `person_email`/`person_phone` outside Create's request
  echo. Step 1's contact-method assertion is therefore **not currently exercisable against the
  real backend** — `PersonHeader`'s email/phone render branches are correct against the
  contract's declared shape and will render real data the moment this backend gap closes; this
  is a pre-existing backend limitation (predates T19), not a frontend defect.

## PO-AC-M1..M5 — merge
9. Click "Merge…", leave the target id empty.
   Expected: Continue is disabled.
10. Enter a valid second person's id, click Continue.
    Expected: a confirm dialog states the target survives, this record is archived, and
    survivor's values win any primary-field conflict — before any request fires.
11. Confirm the merge with a valid, non-archived target.
    Expected: `POST /people/{id}/merge` returns 200; the merge dialog closes.
12. Retry step 9-11 entering this person's own id as the target.
    Expected: 422 `validation_error` — the dialog renders the response's `detail` verbatim (e.g.
    "Cannot merge a person into themself"), not a paraphrase.
13. Retry entering an already-archived/merged-away person's id as the target.
    Expected: 422 `validation_error` renders similarly.
14. From two browser tabs, attempt to merge the *same* loser person into two different targets at
    nearly the same time.
    Expected: the second write's response — inspect its actual `code` in the Network tab. If it
    is `409 code: version_skew`, the dialog renders the concurrent-merge-loss copy. **If it is
    NOT `version_skew`** (a real possibility since `mergePerson` carries no If-Match/version
    param — see PR description), the dialog still renders that other code honestly, never
    claiming `version_skew` didn't happen. [manual] — record which `code` actually came back
    for the PR/ticket record.
