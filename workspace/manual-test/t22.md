# T22 live-UAT — Deal 360

Prereqs: `make infra-up && make migrate-up && make seed-reset && make run` (backend);
`pnpm --dir frontend run dev` (frontend). Log in as a seeded rep who owns at least one open deal
mid-pipeline with 2+ stakeholders (one `economic_buyer`), one closed (won or lost) deal, one deal
with zero stakeholders, and one deal with logged email/call/meeting activities plus at least one
open task assigned to a seeded person. If the default seed lacks any of these, add fixtures to
`scripts/seed`/dev-seed as part of this task (mirroring T21's precedent).

1. **Header + stepper.** Navigate to `/deals/:id` for the mid-pipeline deal.
   Expected: deal name, a "View company" link, value, and a stage stepper showing prior stages
   with a checkmark/muted-done styling, the current stage highlighted, and Won (100%)/Lost (0%)
   nodes present but visually muted (not the deal's actual state).
2. **Explain this number.** Click "Explain this number" under the weighted value.
   Expected: a popover expands showing the literal `value × stage-prob = weighted` arithmetic plus
   "Won = 100%, Lost = 0%"; the win-probability line reads exactly
   "stage default (deterministic) · <stage name>"; clicking the control again collapses it.
3. **Advance, open→open.** Click Advance on the mid-pipeline deal (next stage is also open).
   Expected: advances directly, no confirm dialog, a success toast appears.
4. **Advance into a terminal stage — Won.** Advance a deal whose next stage is terminal; click OK
   in the outcome dialog.
   Expected: deal shows Closed Won, weighted value now reflects 100%, toast mentions "weighted
   100%".
5. **Advance into a terminal stage — Lost, blank reason.** Repeat for a different deal; in the
   dialog take the Cancel/lost-reason path but submit blank.
   Expected: deal stays open, nothing changes, no toast, no server call.
6. **Advance into Lost, with reason.** Repeat, typing a non-blank reason.
   Expected: deal shows Closed Lost with that reason; toast mentions "weighted 0%".
7. **Reopen a closed deal.** Open the closed-deal fixture's page, click Reopen, confirm.
   Expected: deal returns to open status at the pipeline's first open stage; a success toast
   appears; a forced 403 (e.g. via a role without approval authority, if available) renders an
   honest toast naming the cause instead of a silent failure.
8. **Stakeholders rail — multi-threaded.** View the mid-pipeline deal's stakeholders rail.
   Expected: each stakeholder shows a name/title and a role badge (Champion/Stakeholder/Blocker);
   "Multi-threaded" framing shown (stakeholder_count > 1); no "No economic buyer" notice (one is
   seeded).
9. **Stakeholders rail — single-threaded + no economic buyer.** View the zero/one-stakeholder deal.
   Expected: single-threaded risk framing shown when there's exactly one stakeholder (or an honest
   "no stakeholders captured" when there are zero); "No economic buyer identified yet" notice shown
   when no `economic_buyer` row exists.
10. **Activity timeline.** View the deal with logged activities.
    Expected: each email/call/meeting activity shows a timestamp and "via <source_system>"
    provenance; an activity-less deal shows the honest footer "You logged none of this".
11. **Stage-history card.** View any deal's stage-history card.
    Expected: the creation row appears first, followed by each `advance_stage` transition, each
    with a date and a plain-language summary line (e.g. "Devin changed Stage from Discovery to
    Proposal").
12. **Tasks card.** View the deal with a seeded open task.
    Expected: the task shows its assignee's resolved name, an explicit due date, a
    source/captured-by provenance chip, and an unchecked checkbox; clicking the checkbox marks it
    done, shows a success toast, and the row updates to reflect completion. A deal with no tasks
    shows "No tasks yet".
13. **Loading and error states hold per-pane.** Throttle network and reload the page; then stop
    the backend and reload.
    Expected: each pane (header, timeline, history, tasks, stakeholders) shows its own
    skeleton/loading chrome independently; on a backend error, each affected pane renders an
    honest inline error rather than a blank screen or an infinite spinner — one pane's error never
    blocks another pane from rendering its own data.
