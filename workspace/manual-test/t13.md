# T13 Manual UAT — Weighted pipeline roll-up (DEAL-FORM-2 / DEAL-EXT-1)

Run this only after the stack is up with `infra-up -> migrate-up -> seed-reset -> run`.
The commands below assume:

- the API is listening on `http://localhost:8080`
- the database is reachable through `DATABASE_URL`
- you can act as a tenant by sending `X-Workspace-Id` + `X-User-ID`

Use one shell session so the variables set in the bootstrap block stay available for the later steps.

## Bootstrap

```bash
export API_BASE='http://localhost:8080'
export WS_ID='00000000-0000-0000-0000-000000000041'
export USER_ID='55555555-5555-5555-5555-555555555541'
export ROLE_ID='66666666-6666-6666-6666-666666666641'
export PIPELINE_ID='11111111-1111-1111-1111-111111111141'
export STAGE_PROPOSAL_ID='22222222-2222-2222-2222-222222222241'
export STAGE_NEGOTIATION_ID='22222222-2222-2222-2222-222222222242'

psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -c "INSERT INTO workspace (id, name, slug, base_currency) VALUES ('${WS_ID}', 'T13 live UAT workspace', 't13-live-uat', 'EUR') ON CONFLICT (id) DO NOTHING;"
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -c "INSERT INTO pipeline (id, workspace_id, name) VALUES ('${PIPELINE_ID}', '${WS_ID}', 'T13 Weighted Pipeline') ON CONFLICT (id) DO NOTHING;"
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -c "INSERT INTO stage (id, workspace_id, pipeline_id, name, position, semantic, win_probability) VALUES ('${STAGE_PROPOSAL_ID}', '${WS_ID}', '${PIPELINE_ID}', 'Proposal', 1, 'open', 60) ON CONFLICT (id) DO NOTHING;"
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -c "INSERT INTO stage (id, workspace_id, pipeline_id, name, position, semantic, win_probability) VALUES ('${STAGE_NEGOTIATION_ID}', '${WS_ID}', '${PIPELINE_ID}', 'Negotiation', 2, 'open', 80) ON CONFLICT (id) DO NOTHING;"

# RBAC bootstrap: the API authenticates via X-Workspace-Id + X-User-ID, and
# RbacMiddleware loads permissions by joining role/role_assignment for that
# (workspace_id, user_id). Without a real app_user + role + role_assignment row
# here, every request below fails before it reaches the rollup logic.
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -c "INSERT INTO app_user (id, workspace_id, email, display_name) VALUES ('${USER_ID}', '${WS_ID}', 't13-uat@example.com', 'T13 UAT User') ON CONFLICT (id) DO NOTHING;"
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -c "INSERT INTO role (id, workspace_id, key, is_system, permissions) VALUES ('${ROLE_ID}', '${WS_ID}', 't13_admin', true, '{\"pipeline\":{\"read\":{\"row_scope\":\"all\"},\"update\":{\"row_scope\":\"all\"}},\"deal\":{\"read\":{\"row_scope\":\"all\"},\"update\":{\"row_scope\":\"all\"}},\"stage\":{\"read\":{\"row_scope\":\"all\"},\"update\":{\"row_scope\":\"all\"}}}') ON CONFLICT (workspace_id, key) DO UPDATE SET permissions = EXCLUDED.permissions;"
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -c "INSERT INTO role_assignment (workspace_id, role_id, user_id) VALUES ('${WS_ID}', '${ROLE_ID}', '${USER_ID}') ON CONFLICT (role_id, user_id, COALESCE(team_id, '00000000-0000-0000-0000-000000000000'::uuid)) DO NOTHING;"
```

## Step 1 — Worked-example fixture
1. Seeded by the bootstrap block: an EUR-base workspace with pipeline `${PIPELINE_ID}`
   having stage "Proposal" (`${STAGE_PROPOSAL_ID}`, win_probability 60) and
   "Negotiation" (`${STAGE_NEGOTIATION_ID}`, win_probability 80).
2. Create Deal A: `amount_minor=10000000`, `currency=EUR`, stage=Proposal, status=open.
3. Create Deal B: `amount_minor=5000000`, `currency=USD`, stage=Negotiation, status=open.
4. Insert an `fx_rate` row: `USD->EUR`, `rate=0.92`, `rate_date=<today>`.
5. `curl -H "X-Workspace-Id: ${WS_ID}" -H "X-User-ID: ${USER_ID}" "${API_BASE}/pipelines/${PIPELINE_ID}/rollup"`
   Expected: `200`, `unweighted_minor=14600000`, `weighted_minor=9680000`,
   `breakdown` has 2 rows summing exactly to both totals.

## Step 2 — NULL amount deal [auto]
1. Create Deal C in pipeline `${PIPELINE_ID}`, stage Proposal, `amount_minor=null`, `currency=null`,
   status=open.
2. Re-read the rollup.
   Expected: Deal C's breakdown row has `base_value=0`, `weighted_value=0`,
   `no_amount=true`; totals unchanged from Step 1 (Deal C contributes 0, but is present, not
   dropped).

## Step 3 — Closed deal excluded [auto]
1. Create Deal D in pipeline `${PIPELINE_ID}`, `amount_minor=9000000`, `currency=EUR`, then move it to
   `status=won`.
2. Re-read the rollup.
   Expected: Deal D does not appear in `breakdown`; totals are unaffected by its 9,000,000.

## Step 4 — Missing FX rate -> 422 [auto]
1. Create Deal E in pipeline `${PIPELINE_ID}`, stage Proposal, `amount_minor=1000000`, `currency=GBP`, no
   `fx_rate` row seeded for GBP->EUR.
2. Read the rollup.
   Expected: `422`, `code=fx_rate_unavailable`, `details.currency=GBP`, `details.as_of`
   populated (today's date). Delete/close Deal E before continuing to keep later steps clean.

## Step 5 — Multi-currency never cross-sums native units [auto]
1. With Deal A (EUR) and Deal B (USD, rate 0.92) from Step 1 still open, add Deal F:
   `amount_minor=1000000`, `currency=GBP`, stage Negotiation, plus an `fx_rate` row
   `GBP->EUR` rate `1.15`.
2. Read the rollup.
   Expected: `unweighted_minor` includes Deal F's converted `1150000` (1,000,000 * 1.15), not
   its raw `1000000` summed as if it were EUR — verify by checking Deal F's `breakdown` row
   shows `base_value=1150000`.

## Step 6 — Live win_probability read [manual]
1. Note the current `weighted_minor` from Step 1's read.
2. Via `PATCH /stages/${STAGE_PROPOSAL_ID}` (RC-1), set `win_probability=55`.
3. Re-read the rollup immediately (no cache-clear step).
   Expected: Deal A's `weighted_value` recomputes at 55% (5,500,000, down from 6,000,000);
   the change is visible on this read, not delayed.
4. `[manual]` — this step is flagged manual because verifying "not cached" requires a human
   to confirm no intermediate state (e.g. a stale browser tab) shows the old number; the API
   response itself is checked programmatically by `TestRollup_LiveWinProbabilityRead`.

## Step 7 — `make check` [auto]
Run `make check` from the repo root.
Expected: exit 0.
