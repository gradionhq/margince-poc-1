# T17 Manual UAT — restore (un-archive) for person/organization/deal

Prereqs: `make infra-up && make migrate-up && make seed-reset && make run` (API on
`localhost:8080` unless overridden). Every request below needs `X-User-ID: <seeded human user
id — e.g. the seeded admin, 00000000-0000-0000-0010-000000000001>` and a `X-Workspace-Id` header
carrying a valid seeded workspace id — substitute your seed's actual values. `jq` is used to
pretty-print/extract; drop it if unavailable.

## Step 1 — Create + archive a person, then restore it

```bash
PERSON_ID=$(curl -s -X POST localhost:8080/people \
  -H "Content-Type: application/json" -H "X-User-ID: 00000000-0000-0000-0010-000000000001" -H "X-Workspace-Id: $WS" \
  -d '{"full_name":"UAT Restorable Person","source":"ui","captured_by":"human:test"}' | jq -r .id)

curl -s -X DELETE "localhost:8080/people/$PERSON_ID" -H "X-User-ID: 00000000-0000-0000-0010-000000000001" -H "X-Workspace-Id: $WS" | jq .archived_at
# Expected: a non-null ISO timestamp (the person is now archived).

curl -s -o /tmp/t17_person_restore.json -w "%{http_code}\n" -X POST "localhost:8080/people/$PERSON_ID/restore" \
  -H "X-User-ID: 00000000-0000-0000-0010-000000000001" -H "X-Workspace-Id: $WS"
# Expected: 200
cat /tmp/t17_person_restore.json | jq .archived_at
# Expected: null
```

## Step 4 — Restoring a merged organization is refused (PO-AC-18 boundary)

```bash
ORG_SURVIVOR=$(curl -s -X POST localhost:8080/organizations \
  -H "Content-Type: application/json" -H "X-User-ID: 00000000-0000-0000-0010-000000000001" -H "X-Workspace-Id: $WS" \
  -d '{"display_name":"UAT Merge Survivor","source":"ui","captured_by":"human:test"}' | jq -r .id)
ORG_MERGED=$(curl -s -X POST localhost:8080/organizations \
  -H "Content-Type: application/json" -H "X-User-ID: 00000000-0000-0000-0010-000000000001" -H "X-Workspace-Id: $WS" \
  -d '{"display_name":"UAT Merge Away","source":"ui","captured_by":"human:test"}' | jq -r .id)

# There is no wired merge route for organizations yet (pre-existing gap, out of this
# ticket's scope) — seed the merged state directly for this UAT step, same pattern as
# Step 5's deal-archive workaround:
psql "$DATABASE_URL" -c "UPDATE organization SET archived_at=now(), merged_into_id='$ORG_SURVIVOR' WHERE id='$ORG_MERGED';"

curl -s -o /tmp/t17_org_merged_restore.json -w "%{http_code}\n" -X POST "localhost:8080/organizations/$ORG_MERGED/restore" \
  -H "X-User-ID: 00000000-0000-0000-0010-000000000001" -H "X-Workspace-Id: $WS"
# Expected: 422 (merged_into_id set — restore refused, merge reversal is a separate path)
```

## Step 5 — Create + archive a deal, then restore it

```bash
PIPELINE_ID=<seed-known-pipeline-id>   # substitute from your seed data
STAGE_ID=<seed-known-open-stage-id>    # substitute from your seed data

DEAL_ID=$(curl -s -X POST localhost:8080/deals \
  -H "Content-Type: application/json" -H "X-User-ID: 00000000-0000-0000-0010-000000000001" -H "X-Workspace-Id: $WS" \
  -d "{\"name\":\"UAT Restorable Deal\",\"pipeline_id\":\"$PIPELINE_ID\",\"stage_id\":\"$STAGE_ID\",\"source\":\"ui\",\"captured_by\":\"human:test\"}" | jq -r .id)

# There is no wired archive route for deals yet (pre-existing gap, out of this ticket's
# scope) — seed the archived state directly for this UAT step:
psql "$DATABASE_URL" -c "UPDATE deal SET archived_at=now() WHERE id='$DEAL_ID';"

curl -s -o /tmp/t17_deal_restore.json -w "%{http_code}\n" -X POST "localhost:8080/deals/$DEAL_ID/restore" \
  -H "X-User-ID: 00000000-0000-0000-0010-000000000001" -H "X-Workspace-Id: $WS"
# Expected: 200
cat /tmp/t17_deal_restore.json | jq .archived_at
# Expected: null

psql "$DATABASE_URL" -c "SELECT count(*) FROM event_outbox WHERE entity_id='$DEAL_ID' AND topic='deal.restored';"
# Expected: 1
psql "$DATABASE_URL" -c "SELECT count(*) FROM audit_log WHERE entity_id='$DEAL_ID' AND entity_type='deal' AND action='restore';"
# Expected: 1
```

**Note:** the spec's suggested Step 7 (email-collision 409 on person restore) is not
included — confirmed unreachable under the current schema: `uq_person_email` is scoped to
`person_email.archived_at`, never toggled independent of the owning person, so no two live
people can ever hold the same email regardless of archive/restore state (the second `psql`
insert above would itself fail with a unique-violation before any restore call). Same applies
to the org-domain analog (`uq_org_domain` is workspace-absolute, no `archived_at` scoping at
all). See the plan's "Known spec defect" note and each task's per-entity notes.
