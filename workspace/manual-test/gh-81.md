# GH-81 — Manual/Live UAT: Backend Conformance (Data Model, Contract Pipeline, Structural Refactor)

**Feature:** WS-C restores pinned data-model conformance (uuidv7 PKs, `updated_at` triggers,
canonical RLS, restored `session`/`passport`/`app_user` columns, per-workspace
`consent_purpose`). WS-D adds MCP-tool codegen + centralizes tier/approval-gate enforcement
behind `platform/toolgate`, and enables oapi-codegen `ServerInterface` generation. WS-E
standardizes every module onto the `domain/app/ports/adapters/transport` shape and splits the
`directory` catch-all module into `organizations`/`deals`/`activities`/`leads`/`partners`/
`relationships`/`people`/`audithistory`/`datasourcebindings`. None of WS-E's move changes any
HTTP route, request/response shape, or RBAC/RLS behavior — it is a structural refactor, so every
command below must produce byte-identical results before and after the split.

**Step 1** (tier/approval gating) and **Step 2** (CRUD/RLS smoke) are **already true on `main`
today** — the underlying behavior predates this ticket and this guide exercises it as a
regression bar the WS-D/WS-E refactor must not break. **Step 3** (`consent_purpose`
per-workspace) requires Task 1's migration (`000070_ws_c_conformance`) — expect it to fail with a
`column "workspace_id" does not exist` / `column "key" does not exist` error until Task 1 lands.
**Step 4** (build variants) and **Step 5** (full gate) are the whole-ticket DONE criteria.

Run every command from the **repo root** unless a `cd` is specified.

## Step 1 — MCP tool call respects tier/approval gating **[auto]**

Covers AC-D2's acceptance line: an agent-principal call to a 🟡-tier tool without a token → 403;
with a valid single-use token → 200 and the token is consumed (replay fails); a human-principal
call to the same tool → 200 with no token; a dynamic-tier tool (`advance_deal`) resolves tier from
the target stage's semantic (open→open stays 🟢, entering `won`/`lost` becomes 🟡).

### 1a. `merge_records`/person — human bypass, agent gate, single-use token

```bash
make test-it DIR=backend/internal/modules/people/transport \
  RUN='TestMergePersonHumanNoTokenSucceeds|TestMergePersonAgentRequiresApprovalToken'
```

**Expected:** both `--- PASS`. `TestMergePersonAgentRequiresApprovalToken` asserts, in order: no
token → `403` with `code=approval_required`; a `crmapprovals.SignToken`-minted token bound to
`Tool: "merge_records"` + the request's `HashDiff({person_id, target_id})` → `200`; replaying the
identical token against the identical pair → `403` (single-use consumption, not a diff mismatch).

### 1b. `merge_records`/organization — same gate, org handler

```bash
make test-it DIR=backend/internal/modules/directory/transport \
  RUN='TestMergeOrgHumanNoTokenSucceeds|TestMergeOrgAgentRequiresApprovalToken'
```

**Expected:** both `--- PASS` — the org merge handler enforces the identical gate independently.

### 1c. `advance_deal` — dynamic tier resolution + single-use token

```bash
make test-it DIR=backend/internal/modules/directory/transport \
  RUN='TestAdvanceDeal_UAT1_OpenToOpen_NoTokenNeeded|TestAdvanceDeal_UAT2_AgentWithoutToken_403ApprovalRequired|TestAdvanceDeal_UAT3_AgentWithValidToken_SucceedsThenReplayRejected'
```

**Expected:** all three `--- PASS`.
- UAT1: agent principal, `open`→`open` transition, **no token** → `200` (tier resolves 🟢 — the
  dynamic resolver reads the *target* stage's `semantic`, not a hardcoded verb tier).
- UAT2: agent principal, `open`→`won` (target semantic `won` ⇒ 🟡), **no token** → `403` with
  `code=approval_required`; the deal's `status`/`version` are unchanged in the DB (no partial
  mutation).
- UAT3: agent principal, `open`→`won`, a `SignToken`-minted token bound to `Tool: "advance_deal"`
  + `HashDiff({deal_id, to_stage_id, status})` + `TargetVersion` → first use `200` with
  `closed_at`/`fx_rate_to_base` populated; replaying the same token → `403` with
  `code=approval_token_invalid`.

## Step 3 — `consent_purpose` is per-workspace, not global **[auto]**

Requires Task 1's migration (`000070_ws_c_conformance`) to be applied. **Expect this step to fail
with `column "workspace_id" does not exist` (or `column "key" does not exist`) against `main`

(`00000000-0000-0000-0000-000000000001`, backfilled by the migration, D2 step 3) as one tenant and

### 3c. Two workspaces, same key, different ids

```bash
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -c "
  SELECT cp_dev.id AS id_in_dev, cp_new.id AS id_in_new, cp_dev.key
  FROM consent_purpose cp_dev
  JOIN consent_purpose cp_new ON cp_dev.key = cp_new.key
  WHERE cp_dev.workspace_id = '${DEV_WS}' AND cp_new.workspace_id = '${SIGNUP_WS_ID}'
    AND cp_dev.key = 'marketing_email';"
```
**Expected:** one row where `id_in_dev <> id_in_new`, both `key = marketing_email` — the same
logical purpose has a distinct row per workspace (one backfilled from the pre-migration global row,
the other freshly seeded at signup).

### 3d. Consent recorded in one workspace never appears in the other's rows

```bash
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -c "
  WITH cp AS (SELECT id FROM consent_purpose WHERE workspace_id = '${DEV_WS}' AND key = 'marketing_email')
  INSERT INTO person_consent (person_id, workspace_id, purpose_id, state, source, policy_version)
  SELECT '${ALICE_ID}', '${DEV_WS}', cp.id, 'granted', 'uat', 'v1' FROM cp
  ON CONFLICT (workspace_id, person_id, purpose_id) DO UPDATE SET state = EXCLUDED.state;"

psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -c "
  SELECT count(*) FROM person_consent pc
  JOIN consent_purpose cp ON cp.id = pc.purpose_id
  WHERE cp.workspace_id = '${SIGNUP_WS_ID}' AND pc.person_id = '${ALICE_ID}';"
```
**Expected:** `0` — a `person_consent` row keyed to the dev workspace's `consent_purpose.id`
cannot join against the signup workspace's `consent_purpose` rows (different `id`s per D2), so
granting consent in one workspace leaves the other workspace's consent state untouched.
