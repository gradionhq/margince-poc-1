# T07 Manual UAT — Fuzzy Dedupe Scoring (PO-F-1/PO-F-2)

Prereqs: `make infra-up && make migrate-up && make seed-reset && make run` (backend on
`localhost:8080`). Every request needs `X-User-ID: human:018f3a1b-0000-7000-8000-000000000001` (a
seeded workspace admin) and `Content-Type: application/json`.

## Step 1 — Create an org with a domain [auto]
```
curl -s -X POST localhost:8080/organizations \
  -H 'X-User-ID: human:018f3a1b-0000-7000-8000-000000000001' -H 'Content-Type: application/json' \
  -d '{"display_name":"Acme Corp","source":"ui","captured_by":"human:018f3a1b-0000-7000-8000-000000000001","domains":[{"domain":"acme.com","is_primary":true}]}'
```
Expected: `201`, body has `id`, `domains[0].domain == "acme.com"`, `dedupe_review` absent/null (no
prior candidate).

## Step 2 — Create a person at that org [auto]
```
curl -s -X POST localhost:8080/people \
  -H 'X-User-ID: human:018f3a1b-0000-7000-8000-000000000001' -H 'Content-Type: application/json' \
  -d '{"full_name":"John Doe","emails":[{"email":"john@acme.com","email_type":"work","is_primary":true}],"source":"ui","captured_by":"human:018f3a1b-0000-7000-8000-000000000001"}'
```
Expected: `201`, `dedupe_review` absent/null. Note the returned `id` as `<john_id>`.

## Step 2b — Give John a current-primary employment at Acme [auto]
This is what drives the ladder's 1.0 rung in Step 3 below (PLAN-REVIEW B1: without a live
current-primary employment row, a shared email domain alone only reaches the 0.8 rung —
confidence ~0.892, not 0.982). Substitute `<john_id>` and `<acme_org_id>` (from Step 1) below.
```
curl -s -X POST localhost:8080/relationships \
  -H 'X-User-ID: human:018f3a1b-0000-7000-8000-000000000001' -H 'Content-Type: application/json' \
  -d '{"kind":"employment","person_id":"<john_id>","organization_id":"<acme_org_id>","is_current_primary":true,"source":"ui","captured_by":"human:018f3a1b-0000-7000-8000-000000000001"}'
```
Expected: `201`, `is_current_primary: true`.

## Step 3 — Create a near-duplicate name at the same current-primary org -> FUZZY_REVIEW [auto]
```
curl -s -X POST localhost:8080/people \
  -H 'X-User-ID: human:018f3a1b-0000-7000-8000-000000000001' -H 'Content-Type: application/json' \
  -d '{"full_name":"Jon Doe","emails":[{"email":"jon@acme.com","email_type":"work","is_primary":true}],"source":"ui","captured_by":"human:018f3a1b-0000-7000-8000-000000000001"}'
```
Expected: `201` (create still succeeds — never blocks), body carries
`"dedupe_review": {"candidate_id": "<john_id>", "confidence": 0.982}` (± 0.001 — the 1.0 org-match
rung, because Step 2b gave John a confirmed current-primary employment at the same org Jon's email
domain resolves to). Without Step 2b this would read ~0.892 (the 0.8 domain-only rung) instead.

## Step 4 — Exact-email collision is unaffected (still 409, no scoring) [auto]
```
curl -s -o /dev/null -w '%{http_code}\n' -X POST localhost:8080/people \
  -H 'X-User-ID: human:018f3a1b-0000-7000-8000-000000000001' -H 'Content-Type: application/json' \
  -d '{"full_name":"Someone Else","emails":[{"email":"john@acme.com","email_type":"work","is_primary":true}],"source":"ui","captured_by":"human:018f3a1b-0000-7000-8000-000000000001"}'
```
Expected: `409` (unchanged pre-existing behavior — the exact-key tier still 409s before any fuzzy
scoring runs).

## Step 5 — Unrelated name creates plainly, no flag [auto]
```
curl -s -X POST localhost:8080/people \
  -H 'X-User-ID: human:018f3a1b-0000-7000-8000-000000000001' -H 'Content-Type: application/json' \
  -d '{"full_name":"Zara Ng","source":"ui","captured_by":"human:018f3a1b-0000-7000-8000-000000000001"}'
```
Expected: `201`, `dedupe_review` absent/null.

## Step 6 — Org name-only fuzzy tier (PO-F-2) [auto]
```
curl -s -X POST localhost:8080/organizations \
  -H 'X-User-ID: human:018f3a1b-0000-7000-8000-000000000001' -H 'Content-Type: application/json' \
  -d '{"display_name":"Acme GmbH","source":"ui","captured_by":"human:018f3a1b-0000-7000-8000-000000000001"}'
```
Expected: `201`, `"dedupe_review": {"candidate_id": "<acme_corp_id_from_step_1>", "confidence": 1.0}`
("Acme Corp" and "Acme GmbH" both normalize to "acme" after legal-suffix strip).

## Step 7 — Empty full_name skips the fuzzy tier entirely [manual]
No client-reachable path creates a person with an empty `full_name` (`POST /people` 400s on
`missing_required_fields` at the handler edge before the store is ever called) — this edge case is
proven at the store layer by `TestPersonCreate_EmptyFullName_SkipsFuzzyTier`
(`store_person_dedupe_test.go`), not exercisable live through the HTTP surface. Confirm by reading
that test, not by a live request.
