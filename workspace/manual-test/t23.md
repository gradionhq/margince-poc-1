# T23 Manual UAT - Archive/Restore UX + STATE-1..5 sweep

Run this against the running stack, from the frontend UI in a browser:

```bash
make infra-up && make migrate-up && make seed-reset && make run
# in a second shell
cd frontend && pnpm dev
```

The steps below assume:

- the frontend is reachable at `http://localhost:5173`
- the API is listening on `http://localhost:8080`
- `make seed-reset` has populated at least one live contact, one live company, and one live
  deal, and you know their names/ids for navigating the UI
- you are logged in as a user with full read/write permissions unless a step says otherwise

## Step 1 (AC1): Archive a contact from the contacts list row menu

1. Open the contacts list.
2. Open the row menu for a live (non-archived) contact and choose Archive.
3. Confirm in the dialog.

Expected: a success toast appears; the row disappears from the live list; `GET /people/{id}`
for that contact still returns `200 OK` with `archived_at` set (verify via curl or devtools
Network tab).

## Step 2 (AC1/AC2): Archive a contact from its 360 page

1. Open a different live contact's 360 page.
2. Use the Archive… button and confirm.

Expected: a success toast appears; the page does not redirect to a 404 or go blank; a
persistent "This record is archived" banner appears on the page with a Restore action.

## Step 3 (AC2): Restore the contact from Step 2

1. On the archived contact's 360 page, click Restore.

Expected: a success toast appears; the archived banner disappears; the record reappears in
the live contacts list.

## Step 4 (AC1/AC2): Repeat Steps 1-3 for a company

1. Archive a live company from the companies list row menu.
   Expected: toast, row disappears from the live list, `GET /organizations/{id}` still
   returns it with `archived_at` set.
2. Archive a different live company from its 360 page's Archive… button.
   Expected: banner appears with Restore, no 404/blank.
3. Restore that company.
   Expected: banner disappears, record reappears in the live companies list.

## Step 5 (AC1/AC2): Repeat Steps 1-3 for a deal, across both pipeline views

1. Archive a live deal from the pipeline board view.
   Expected: toast, card disappears from the live board.
2. Archive a different live deal from the pipeline table view.
   Expected: toast, row disappears from the live table.
3. Archive a third live deal from its 360 page's Archive… button.
   Expected: banner appears with Restore, no 404/blank.
4. Restore that deal.
   Expected: banner disappears, record reappears in the live pipeline board/table.

## Step 6 (AC2): Restore 409 renders the existing-record pointer

1. Using two live people that share an email (seed or create one), archive one of them.
2. Attempt to Restore the archived person.

Expected: the restore is refused (409, dedupe-refused per T17's live-scoped collision
check); the UI renders a link/reference to the colliding live record (the existing-record
pointer), never a generic error toast and never a silent failure. Clicking the pointer
navigates to the colliding live record's 360 page.

## Step 7 (AC3): STATE-1..5 sweep across all six screens

Repeat for each of: contacts list, person 360, companies list, company 360, pipeline
board/table, deal 360.

1. **STATE-1 (honest empty)**: view the screen/panel with zero rows (e.g. a fresh workspace
   or a filter matching nothing). Expected: a real "nothing here yet" message, no fabricated
   counts or placeholder rows.
2. **STATE-2 (skeleton loading)**: throttle the network (devtools) and reload. Expected:
   page chrome renders immediately; content streams in via a skeleton/loading state within
   budget, not a blank page.
3. **STATE-3 (error card)**: stop the backend mid-request (e.g. kill `make run` after the
   page starts loading, or block the API host in devtools). Expected: an error card with
   cause + retry action renders in place of the affected content; the whole screen never
   blanks.
4. **STATE-5 (nothing-grounded panels absent)**: with devtools element inspection, confirm
   the warm signal, firmographics, and AI content panels are absent from the DOM entirely
   (no node, no "coming soon" stub) — not merely CSS-hidden.
