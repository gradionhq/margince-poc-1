# T06 Live UAT Guide — Person Merge + Person-Email Create-Time Dedupe

Run this only after the stack is up:

```bash
make infra-up && make migrate-up && make seed-reset && make run
```

The commands below assume:

- the API is listening on `http://localhost:8080`
- the database is reachable through `$DATABASE_URL`
- Python 3 is available (used to decode JSON response bodies and to mint the
  `X-Approval-Token` value in step 5)
- you run all steps in one shell session so exported variables stay available

## Prerequisites: `APPROVAL_TOKEN_SIGNING_SECRET`

Step 5 mints an `X-Approval-Token` for an agent principal. The signing seam
(`backend/internal/modules/approvals/token.go`) reads this env var at runtime, so it must
be set **both** where `make run` runs (the server) and in this UAT shell (so the minted
token's signature matches). Add it to `.env.local` (or export it) before starting the
server:

```bash
# in .env.local (or export before make run)
APPROVAL_TOKEN_SIGNING_SECRET=dev-t06-uat-secret
```

Then in this UAT shell:

```bash
export APPROVAL_TOKEN_SIGNING_SECRET=dev-t06-uat-secret
```

Any non-empty string works for dev as long as it matches on both sides.

## Step 1: `POST /people` twice with the same email → 409 `duplicate_email` (PO-AC-16)

```bash

  -H "Content-Type: application/json" \
  -H "X-Workspace-Id: ${WS_ID}" \
  -H "X-User-ID: ${ADMIN_USER_ID}" \
```

```bash
export PERSON_A="$(psql "$DATABASE_URL" -At -c \
```

## Step 2: `POST /people/{A}/merge {"target_id":"B"}` as a human → 200, non-lossy relink

Create the target person B:

```bash
export PERSON_B="$(curl -sS -X POST \
  -H "Content-Type: application/json" \
  -H "X-Workspace-Id: ${WS_ID}" \
  -H "X-User-ID: ${ADMIN_USER_ID}" \
  -d '{"full_name":"Dana Target","emails":[{"email":"dana.target@t06-uat.example","email_type":"work","is_primary":true}],"source":"uat","captured_by":"human:t06"}' \
  "${API_BASE}/people" | python3 -c "import sys,json; print(json.load(sys.stdin)['id'])")"
echo "PERSON_B=${PERSON_B}"
```

Expected: `201 Created` (implicit in a non-empty `PERSON_B`).

Now merge A into B as a human principal (no `X-Approval-Token` header — a human's direct
call is itself the approval):

```bash
curl -i -sS -X POST \
  -H "Content-Type: application/json" \
  -H "X-Workspace-Id: ${WS_ID}" \
  -H "X-User-ID: ${ADMIN_USER_ID}" \
  -d "{\"target_id\":\"${PERSON_B}\"}" \
  "${API_BASE}/people/${PERSON_A}/merge"
```

Expected: `200 OK`, body `"id"` equals `${PERSON_B}`.

`GET /people/{id}` (`PersonStore.Get`, `backend/internal/modules/directory/store.go`)
filters `WHERE ... AND archived_at IS NULL` — a pre-existing behavior that predates T06
(`git blame` on that line points at the original person-feature commit, not this branch)
and contradicts `crm.yaml`'s `getPerson` doc ("Fetchable by id even when archived"). Since
A is now archived, `GET /people/${PERSON_A}` returns `404`, not `200` — out of scope for
this ticket to fix (flag it separately if not already tracked). Verify the merge outcome
directly instead, the same way Task 2's own integration test does:

```bash
psql "$DATABASE_URL" -c \
  "SELECT merged_into_id, archived_at IS NOT NULL AS archived FROM person WHERE id='${PERSON_A}';"
```

Expected: `merged_into_id` equals `${PERSON_B}` and `archived` is `t`.

**[auto]** A's pre-existing employment relationship being relinked to point at B
(`person_id: B`, zero orphaned FKs) cannot be driven live: this pruned skeleton tree
registers only `/people`, `/organizations`, `/deals`, `/pipelines`, `/stages`, and
`/partners` in `backend/cmd/api/routes.go` (`registerCoreCRUD`) — there is no
`/relationships` handler wired (confirmed via
`grep -rln "RelationshipHandler" backend/internal` returning nothing), and
`GET /organizations/{id}`'s response does not populate a `relationships` field either
(confirmed via `grep -n "Relationships" backend/internal/modules/directory/store_org.go`
returning nothing) — so there is no live read to point a curl at. This is covered instead
by the integration test
`TestPersonMergeRelinksEmailsPhonesRelationshipsAndActivityLinks`
(`backend/internal/modules/directory/store_merge_person_test.go`), which asserts the
relationship row's `person_id` is updated to the survivor with zero orphaned FKs.

## Step 5: Agent principal — approval gate on `merge_records`

Mint a passport so the middleware sets `IsAgent: true` on the request:

```bash
export AGENT_TOKEN=$(curl -sS -X POST \
  -H "Content-Type: application/json" \
  -H "X-Workspace-Id: ${WS_ID}" \
  -H "X-User-ID: ${ADMIN_USER_ID}" \
  -d '{"scopes":[],"expires_in_seconds":3600}' \
  "${API_BASE}/passports" | python3 -c "import sys,json; print(json.load(sys.stdin)['token'])")
echo "AGENT_TOKEN=${AGENT_TOKEN}"
```

Expected: `POST /passports` returns `201 Created`; `AGENT_TOKEN` is non-empty.

Create a fresh pair for this scenario (A2 merges into B2), then have the agent attempt
the merge with no `X-Approval-Token`:

```bash
export PERSON_A2="$(curl -sS -X POST \
  -H "Content-Type: application/json" \
  -H "X-Workspace-Id: ${WS_ID}" \
  -H "X-User-ID: ${ADMIN_USER_ID}" \
  -d '{"full_name":"T06 Agent Loser","emails":[{"email":"agent-loser@t06-uat.example"}],"source":"uat","captured_by":"human:t06"}' \
  "${API_BASE}/people" | python3 -c "import sys,json; print(json.load(sys.stdin)['id'])")"
export PERSON_B2="$(curl -sS -X POST \
  -H "Content-Type: application/json" \
  -H "X-Workspace-Id: ${WS_ID}" \
  -H "X-User-ID: ${ADMIN_USER_ID}" \
  -d '{"full_name":"T06 Agent Target","emails":[{"email":"agent-target@t06-uat.example"}],"source":"uat","captured_by":"human:t06"}' \
  "${API_BASE}/people" | python3 -c "import sys,json; print(json.load(sys.stdin)['id'])")"

curl -i -sS -X POST \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer ${AGENT_TOKEN}" \
  -d "{\"target_id\":\"${PERSON_B2}\"}" \
  "${API_BASE}/people/${PERSON_A2}/merge"
```

Expected: `403 Forbidden`, body `{"code":"approval_required",...}`. Confirm no mutation:

```bash
psql "$DATABASE_URL" -c "SELECT merged_into_id, archived_at FROM person WHERE id='${PERSON_A2}';"
```

Expected: `merged_into_id` and `archived_at` both `NULL`.

Mint a single-use `X-Approval-Token` bound to this exact merge. The token's `diff_hash`
claim must match `crmapprovals.HashDiff({"person_id": <A2>, "target_id": <B2>})`
(`backend/internal/modules/people/transport/handler_person.go`'s merge path), and its
`tool` claim must be the contract's declared `x-mcp-tool` verb, `merge_records`
(`crm.yaml:335`) — never a per-entity string like `merge_person`:

```bash
export APPR_TOKEN=$(python3 - "${PERSON_A2}" "${PERSON_B2}" "uat5-jti-${PERSON_A2}" <<'PYEOF'
import hmac as _hmac, hashlib, base64, json, os, sys
from datetime import datetime, timezone, timedelta

def b64u(b):
    return base64.urlsafe_b64encode(b).rstrip(b'=').decode()

root = os.environ['APPROVAL_TOKEN_SIGNING_SECRET']
ws_id = os.environ['WS_ID']
person_id, target_id, jti = sys.argv[1], sys.argv[2], sys.argv[3]

# Per-workspace key: HMAC-SHA256(root, workspace_id)
key = _hmac.new(root.encode(), ws_id.encode(), hashlib.sha256).digest()

# diff_hash: SHA-256 of the canonical (sort_keys) JSON of the bound fields —
# must match crmapprovals.HashDiff's field set exactly (person_id, target_id).
diff = {'person_id': person_id, 'target_id': target_id}
diff_hash = b64u(hashlib.sha256(
    json.dumps(diff, sort_keys=True, separators=(',', ':')).encode()
).digest())

exp = (datetime.now(timezone.utc) + timedelta(minutes=5)).strftime('%Y-%m-%dT%H:%M:%SZ')
claims = {'jti': jti, 'approval_id': 'uat-' + jti, 'workspace_id': ws_id,
          'tool': 'merge_records', 'diff_hash': diff_hash, 'exp': exp, 'single_use': True}

hdr = b64u(json.dumps({'alg': 'HS256', 'typ': 'JWT'}, separators=(',', ':')).encode())
pay = b64u(json.dumps(claims, separators=(',', ':')).encode())
sig = _hmac.new(key, (hdr + '.' + pay).encode(), hashlib.sha256).digest()
print(hdr + '.' + pay + '.' + b64u(sig))
PYEOF
)
echo "APPR_TOKEN=${APPR_TOKEN}"

# First use: agent merges with a valid token.
curl -i -sS -X POST \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer ${AGENT_TOKEN}" \
  -H "X-Approval-Token: ${APPR_TOKEN}" \
  -d "{\"target_id\":\"${PERSON_B2}\"}" \
  "${API_BASE}/people/${PERSON_A2}/merge"
```

Expected: `200 OK`, body `"id"` equals `${PERSON_B2}`.

Reuse the same token (identical request, already-merged pair):

```bash
curl -i -sS -X POST \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer ${AGENT_TOKEN}" \
  -H "X-Approval-Token: ${APPR_TOKEN}" \
  -d "{\"target_id\":\"${PERSON_B2}\"}" \
  "${API_BASE}/people/${PERSON_A2}/merge"
```

Expected: `403 Forbidden`, body `{"code":"approval_token_invalid",...}` — the jti was
already consumed (single-use), so this proves replay-rejection, not just a stale
`already_merged` re-check.
