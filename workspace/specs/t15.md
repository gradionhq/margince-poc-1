# T15 — Backend — partner registration: upsert/get/list on the aligned schema + the S-E03.6 deal-side slice

## Goal

Build the `partner` HTTP transport layer from scratch (`upsertPartner`, `getPartner`,
`listPartners`) and wire the deal-side S-E03.6 slice: `createDeal`/`updateDeal` accept
`partner_org_id`, with attribution changes audited. `crm.yaml` already fully declares every
operation/schema this ticket needs (landed by T02's PO-EXT-5 contract alignment) — this is
transport + store + one migration, no contract change.

## Discovered current state (read before implementing — corrects the ticket body's stated layout)

- **No partner store or handler exists yet** — greenfield. Handler package convention is
  `backend/internal/modules/directory/transport` (where `handler_deal.go`/`handler_org.go`
  already live, per T11/T05 — **not** `modules/people/transport` as the raw ticket body
  says; follow this spec, not the raw body). Store code (a `PartnerStore`, `crmcore`
  package) belongs beside `store_org.go`/`store_deal.go` in
  `backend/internal/modules/directory`.
- **The `partner` table already exists** (migration `000007_partner_relationship.up.sql`)
  but its `CHECK` constraints are **stale relative to PO-DDL-6 and the already-aligned
  contract**:
  - DB `cert_status` CHECK allows `('pending','active','suspended','expired')`; PO-DDL-6 /
    the current `crm.yaml` `Partner.cert_status` enum target values are
    `applied`/`certified`/`suspended` (contract additionally unions in legacy
    `prospect`/`in_certification`/`churned` per T02's additive-first gate decision, but
    those legacy values were never actually written to this table — it's empty, no partner
    row has ever been created). Realign the DB CHECK to PO-DDL-6's
    `('applied','certified','suspended')` directly — do not carry the wrong placeholder
    values (`pending`/`active`/`expired` were never correct; this is not the same
    situation as T02's live-data-carrying deprecation case).
  - DB `margin_tier` CHECK allows `('standard','silver','gold','platinum')`; PO-DDL-6 /
    `crm.yaml` target is `tier1_15`/`tier2_20`/`tier3_25`. Same fix: realign directly.
  - `certified_staff` is `integer`; PO-DDL-6 pins `smallint` — align while touching this
    table (non-breaking, no app code reads the Go type differently).
  - Confirm via `SELECT count(*) FROM partner` in your migration testing that the table is
    genuinely empty in every seeded/test dataset before altering the CHECKs directly (no
    migration-time data rewrite needed if so — if you find seeded rows using the old
    values, backfill-map them to the nearest PO-DDL-6 value in the same migration, then
    apply the CHECK).
- **`deal.partner_org_id` and `idx_deal_partner_org` already exist** (migrations `000003`,
  `000010`); `DealStore.Create`/`Get`/`List` (`store_deal.go`) already read/write/filter
  `PartnerOrgID` — **but the HTTP layer never wires it**:
  - `handler_deal.go`'s `create()` request-body struct has no `partner_org_id` field, so
    `createDeal` can never set it today.
  - `DealStore.Update`'s SQL (`store_deal.go` ~L196-211) has no `partner_org_id` column in
    its `UPDATE ... SET` list, so `updateDeal` (PATCH, map-based) can never change it
    today either.
  - Neither path writes an `audit_log` row for a `partner_org_id` change (Deal `Update`
    writes no audit row at all today — that's a pre-existing gap wider than this ticket;
    scope your audit fix narrowly to emitting one `audit_log` row + one event specifically
    when `partner_org_id` changes via `Update`, per S-E03.6's "attribution changes are
    audited without losing the prior value" — do not attempt a general Deal-Update audit
    fix, that's separate follow-up territory).
- `backend/api/crm.yaml` already declares `POST /organizations/{id}/partner` (upsert),
  `GET /organizations/{id}/partner` (get), `GET /partners` (list, filters
  `partner_role`/`cert_status`), the `Partner`/`UpsertPartnerRequest` schemas, and
  `listDeals`' `partner_org_id` filter param — confirm nothing here needs touching.

## Acceptance criteria (build-verifiable)

1. Migration: realign the `partner` table's `cert_status`/`margin_tier` CHECK constraints
   to PO-DDL-6's target enum values (see above); `certified_staff` becomes `smallint`.
   (verify: integration)
2. `upsertPartner` (`POST /organizations/{id}/partner`): creates or updates the 1:1
   `partner` row for the given `organization_id` (`UNIQUE` constraint already enforces
   1:1 — treat a conflict as an update path, not a 409). Requires the RBAC `admin`
   permission (check the existing RBAC-permission plumbing other 🟡/admin-gated handlers
   use, e.g. wherever `advanceDeal` or similar checks a permission — mirror that idiom, do
   not invent a new one). On success, sets the extended organization's `classification` to
   `'partner'` in the same tx (never duplicates firmographics — only the `classification`
   column is touched on `organization`, no other org fields). (verify: integration)
3. `getPartner` (`GET /organizations/{id}/partner`): 404 (`not_found`) when the org has no
   live `partner` row. (verify: integration)
4. `listPartners` (`GET /partners`): filters by `partner_role` and `cert_status` per
   DM-VOCAB-6 against the PO-DDL-6 enum values, cursor-paginated per the existing
   `Cursor`/`Limit` convention other list endpoints use. (verify: integration)
5. `margin_tier` and `partner_role` are orthogonal, independently settable fields —
   `margin_tier` is **never** DB-constrained against `certified_staff`/`retention_rate`
   (that gating, if any, is application-logic-only and out of this ticket's acceptance
   criteria to enforce beyond accepting the field); tier changes go through the same
   audit path as any other `upsertPartner` mutation. (verify: integration)
6. S-E03.6 deal-side slice: `createDeal` accepts `partner_org_id` in its request body and
   persists it (wire the missing struct field — the store already supports it);
   `updateDeal` (PATCH) accepts and persists a `partner_org_id` change (wire the missing
   `UPDATE` column — the store's `Update` SQL needs the new `SET` clause); a
   `partner_org_id` change via `updateDeal` writes exactly one `audit_log` row capturing
   the prior and new value (via `crmaudit.WriteTx`, mirroring `Create`'s pattern in the
   same file) and enqueues one `deal.updated`-class outbox event (do not invent a new
   event topic if `deal.updated` doesn't exist yet — check `store_deal.go`/consts for the
   existing deal event-topic vocabulary and reuse it; if none exists for generic updates,
   use `deal.partner_assigned` as the topic name). `listDeals`' existing `partner_org_id`
   filter (already wired per T11) needs no further change — confirm with an integration
   test that it round-trips a deal created/updated via this ticket's new fields. (verify:
   integration)
7. Provenance + audit floor: `partner` rows carry non-null `source`/`captured_by`
   (`requireProvenance`-style guard, mirroring `OrgStore.Create`); every `partner` mutation
   (`upsertPartner`'s create and update paths) writes one `audit_log` row + one event
   (`partner.created`/`partner.updated` — pick names consistent with the
   `organization.*`/`deal.*` topic naming convention already in use). (verify: integration)
8. `make check` is green; no regression in existing `directory`/`transport` package tests.
   (verify: command)

## UAT checklist

1. `POST /organizations/{id}/partner` with `{partner_role: "hosting", cert_status:
   "applied", source: ..., captured_by: ...}` on a non-partner org.
   Expected: 201/200, org's `classification` becomes `partner`, one `audit_log` row, one
   `partner.created` outbox row.
2. `POST /organizations/{id}/partner` again on the same org with an updated
   `cert_status: "certified"`. Expected: 200, same partner row updated (not duplicated),
   one `audit_log` row for the update.
3. `GET /organizations/{id}/partner` on an org with no partner row. Expected: 404.
4. `GET /partners?partner_role=hosting&cert_status=certified`. Expected: 200, only
   matching live partner rows.
5. `POST /deals` with `partner_org_id` set to a partner org's id. Expected: 201, response
   includes the persisted `partner_org_id`.
6. `PATCH /deals/{id}` changing `partner_org_id` to a different partner org.
   Expected: 200, new value persisted, one `audit_log` row capturing the prior value.
7. `GET /deals?partner_org_id={id}` returns the deal from steps 5/6.
8. `make check` exits 0; `make test-integration` shows 0 skips on the affected packages.
