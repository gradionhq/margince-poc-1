# ADR-0022 — Build the capture semantics + retrieval engine; borrow only in-boundary transport; capture is memory-first

**Status:** Accepted (Lars, 2026-06-17). Research parent: [`../../research/r3-capture-and-retrieval.md`](../../research/runtime/r3-capture-and-retrieval.md) (closes BACKLOG R3). **Sharpens** P5 (auto-capture) and P7 (own-your-data), and **composes with** [ADR-0020](ADR-0020-customer-supplied-inference.md) (inference is customer-supplied) + [ADR-0021](ADR-0021-relational-pgvector-graph-trigger.md) (relational + pgvector is the substrate). Reinforces, does not unwind, the egress location ladder ([ADR-0008](ADR-0008-lead-object-and-promotion.md) is the lead model; the egress posture is A8 revised / [ADR-0011](ADR-0011-consent-and-retention.md), A24) and the one-governed-surface model ([ADR-0013](ADR-0013-one-governed-surface-and-auth.md)).

## Context

R3 was the last open AI-substrate question: for email/calendar **and messaging (WhatsApp, Telegram)** capture, and for retrieval over it, do we build or borrow — and where is the line that keeps own-your-data (P7) intact? `07-risks.md` R-C3 flagged it needed a capture/retrieval spike.

R3 measured the bar against the **new AI-native CRMs** (Day.ai, Attio, Lightfield, Folk), not the legacy incumbents. They have converged on one architecture — **memory-first auto-capture → context graph → conversational RAG → MCP agent surface** — and share one compromise: your communications live in *their* cloud (every AI-native peer is SaaS-hosted; the only own-your-data peer, Twenty, is not memory-first/AI-native). The legacy pattern is the same failure at larger scale (Salesforce Einstein Activity Capture keeps email in AWS, not your org, 6-month default retention; Gong on AWS for years). Margince already holds the pieces to match the cohort — the context graph (ADR-0007), the `raw jsonb` memory column + provenance (`data-model.md §1.6`), pgvector hybrid retrieval (ADR-0021), and the governed MCP surface (ADR-0009/0013). The decision below names which parts we build, which we borrow, and the invariant that makes the own-your-data square — which no competitor occupies — ours.

## Decision

**Margince builds the capture *semantics* and the retrieval *engine*; it borrows only transport, and only inside the customer's trust boundary. Capture is memory-first, layered on the relational core.**

1. **Capture posture is memory-first.** The raw captured payload (`raw jsonb`, `data-model.md §1.6`) is the **source of truth**; AI derives entities, links, and fields on top of it and may **backfill retroactively** when the model or schema improves (the schema-less-memory move Day.ai/Lightfield are built on). The relational core — real FKs, the typed `relationship` table, provenance, consent/retention — stays the governed, exportable system of record. Memory-first does **not** replace the relational model; it sits beneath the derived rows.
2. **Capture semantics are built.** Normalization, entity resolution / dedupe (`formulas-and-rules.md §1.1`), provenance stamping, context-graph linking, and the raw→derived→backfill pipeline are core differentiators, built in `crm-capture`. No third party owns this layer.
3. **Transport is borrow-or-build, always in-boundary.** No Margince-operated and no CRM-vendor-operated cloud sits in the capture path. Email/calendar: direct provider clients by default (Gmail API + Pub/Sub watch, Microsoft Graph delta + change notifications, IMAP/JMAP, CalDAV), with a self-hostable engine (EmailEngine-class) as an allowed swap. Messaging: the **customer's own credentials** — Telegram TDLib/MTProto self-hosted (in-boundary), WhatsApp Cloud API on the customer's own WhatsApp Business Account.
4. **Cloud-proxy aggregators are rejected.** Nylas / Unipile / Aurinko / Merge — which sync customer comms through the vendor's servers — are **not the default and are forbidden for dedicated/sovereign deployments**. The one permitted exception is an explicit opt-in for the **SaaS multi-tenant tier** (`06 §6.3`) with a residency warning at connect — and it must be an ADR-recorded choice, never a silent default.
5. **Messaging carries an operator-in-path caveat — WhatsApp is not sovereign.** For messaging networks the operator is unavoidably in the path (intrinsic, not our choice). Telegram can be made in-boundary (self-hosted client). **WhatsApp cannot be zero-egress** — the On-Premises API was sunset (2025-10-23); Cloud API runs on Meta's infrastructure. Best achievable is the customer's own WABA + EU data-residency-at-rest + webhook to a customer-hosted endpoint. Any "sovereign / zero-egress" claim **must carve out WhatsApp**.
6. **Retrieval is built on Postgres; embeddings/rerank are customer-supplied.** Hybrid `tsvector` (optionally ParadeDB `pg_search` BM25) + `pgvector` (optionally `pgvectorscale`) fused with Reciprocal Rank Fusion, in `crm-search` — no external vector DB (confirms ADR-0021). Embeddings and an optional cross-encoder reranker run on customer-supplied inference (ADR-0020): self-hostable open models by default (bge-m3 per A9, a self-hosted reranker), BYOK optional; no bundled embedding/rerank service.

## Consequences

- **Positive:** opens the own-your-data square no AI-native competitor occupies — the full memory-first capture + conversational-RAG experience with data in the customer's Postgres and inference customer-supplied; the build-vs-borrow line falls exactly on the differentiator (semantics + memory + retrieval = ours; commodity plumbing = swappable, in-boundary); retroactive backfill (a Day.ai signature feature) drops out of the memory-first posture for free; no new vendor dependency and no new datastore (reuses ADR-0021 substrate, ADR-0020 inference model).
- **Negative / costs:** **memory-first storage growth** — keeping `raw` for every message plus derived rows is heavier than relational-only (mitigated: `raw` is off the hot path / un-indexed per `data-model.md §1.6`, blobs go to S3/MinIO per B7, retention can age `raw` out per A22); **backfill inference cost** — "populate this field from all past email" is a re-derivation pass the customer pays for (mitigated: explicit async budgeted job under the `03b` guardrails, cheap/local-first routing); **WhatsApp is a permanent sovereignty exception** that marketing/docs must state honestly; **building transport** (even borrowing EmailEngine-class) is more work than adopting Unipile, accepted as the cost of P7.
- **Open questions (→ `07-risks.md` R-C3 / R-C7):** which local embedder/reranker meet retrieval quality at acceptable latency (the residual R-C3 item, validated by the spike on both bindings per A23); the memory-first storage envelope per `06 §6.7` tier; whether the optional cross-encoder reranker earns its latency (decided by recall@k in the spike) — all `crm-capture`/`crm-search`-WP details, not load-bearing here.

## What this does NOT change (guardrails)

- The `person` / `organization` / `deal` / `activity` + `activity_link` relational model (`data-model.md`, B6) and the provenance invariants (`source`/`captured_by` NOT NULL, `features/02` AC1.2/ACX.1) — unchanged; memory-first sits *beneath* them, it does not remove them.
- The context-graph capability and its V1 scope ([ADR-0007](ADR-0007-context-graph-is-v1-substrate.md)) and the relational + pgvector substrate ([ADR-0021](ADR-0021-relational-pgvector-graph-trigger.md)) — unchanged; this confirms them as the retrieval engine.
- Customer-supplied inference ([ADR-0020](ADR-0020-customer-supplied-inference.md), A28) — reinforced; embeddings and rerank are inference and are customer-supplied like everything else.
- The egress location ladder / EU-residency default (A8 revised, A24/ADR-0011), secret-stripping hygiene, and the sovereign zero-egress conformance test (`features/02` ACX.6) — reinforced; the in-boundary transport rule is the capture-side expression of the same posture (with WhatsApp the named exception).
- The confirm-first outbound gate (`features/02` ACX.3) — unchanged; capturing inbound messaging is 🟢, sending on a channel stays 🟡.
