# ADR-0003 — BYO-Agent orchestration plus a bundled baseline AI tier

**Status:** Accepted (ratified 2026-06-04; status normalized at vendoring — see README.md). **Superseded in part by [ADR-0020](ADR-0020-customer-supplied-inference.md) (2026-06-16):** the "Layer 2 baseline AI is bundled in the license, not metered" funding clause is retired — the Layer 2 *features* survive, but all inference is customer-supplied (BYOK or self-hosted), paid to the provider directly; Gradion provides none. Read the Layer 2 bullet and its cost-center consequence below through ADR-0020.

## Context

The CRM market's dominant AI failure mode is bolted-on sales agents (70–80% year-one churn) sold as expensive per-seat add-ons on data models never designed for them. Day.ai/Attio/Clarify build proprietary AI; incumbents resell "intelligence" by the seat and are structurally disincentivized to let a client's own agent in. The #1 incumbent pain is total cost, and forced per-seat AI compounds it.

Gradion's principles point a different way: P6 (embrace the LLMs, don't fight them — orchestrate the user's own agents; add only the baseline AI the agentless majority genuinely need; never reimplement what frontier labs do better), P12 (governance designed in — audit, provenance, approval gates, rollback), P7 (own your data, incl. local LLM inference). We need a foundational decision on what AI we build vs. orchestrate.

## Decision

A three-layer AI architecture; we win on *where intelligence runs* and *what it's allowed to do*, not on model intelligence.

- **Layer 1 — BYO-Agent orchestration (foundational).** The CRM is the system of record and action; the user brings the intelligence (their Claude/Cursor/Copilot/self-hosted agent). We expose CRM data and actions as **governed MCP tools** (`crm-agents`): `search_records`, `read_record`, `create/update_record`, `advance_deal`, `draft_email`, `send_email` (gated), `create_task`, `log_activity`, `run_report`, `enrich` — versioned, contract-bound. An **"Agent Seat Passport"** binds each agent connection to a CRM identity with explicit scopes (read-only → draft-only → act-with-approval → auto-execute-low-risk) mapped to RBAC; an agent can never exceed the human's permissions. **Two-tier autonomy:** 🟢 auto-execute low-risk/reversible actions; 🟡 confirm-first for outbound/irreversible/high-value. Dry-run + diff preview; full provenance/audit on every action.
- **Layer 2 — Built-in baseline AI (for the agentless majority).** [ADR-0020: revised funding.] The *features* ship in the product — auto-capture & enrichment (P5, the highest-value feature), summarization, draft assistance (never auto-send at baseline), NL search & reporting, next-best-action hints — but the **inference is customer-supplied** (their own provider key, or a model they self-host), paid to the provider directly; Gradion bundles, meters, and funds nothing. The tiered routing, batching/caching/embedding reuse, and per-workspace budget guardrails with graceful degradation (never blocks core CRM) are retained as **customer-side cost control**, not Gradion margin protection.
- **Layer 3 — Shared substrate.** Auto-capture pipeline (`crm-capture`, River), hybrid retrieval (Postgres tsvector + pgvector; graph store deferred to an ADR after a spike), a provider-agnostic model `Client` (Anthropic/OpenAI/Gemini/local — model choice is config, not architecture; enables local-only inference for P7), and a secret-stripper on egress (A8 revised — privacy is the location ladder, not PII pseudonymization).

Validated by the Stage-6 BYO-agent probe: Claude via MCP performs "find stalled deals, draft follow-ups, queue for approval" under scopes and approval gates, fully audited. If it can't, the design is wrong.

## Consequences

- **Positive:** attacks the #1 incumbent pain (cost) structurally — clients are never forced into per-seat AI to get a working AI-native CRM; an open-source, MCP-native, own-your-data CRM is the natural home for BYO agents (a position incumbents can't take); one coherent agent story across build-time source customization (ADR-0002) and runtime action; governance/provenance is the EU AI Act / GDPR answer (P12); local inference satisfies regulated/data-sensitive clients (P7).
- **Negative / costs:** [ADR-0020: revised — Layer 2 is no longer a Gradion cost center; inference is customer-supplied, so the COGS risk moves to the customer's provider bill.] AI features require the customer to configure a key or self-hosted endpoint (a first-run step; the non-AI core works without one); feature quality varies with the customer's chosen model, so per-task minimum-capability guidance + eval gates + graceful degradation are load-bearing; the MCP tool surface is a security/governance attack surface that must be rigorously scoped and audited.
- **Open questions (→ `07-risks.md`):** pgvector-only vs. add a graph store (**resolved — ADR-0021/R2: relational + pgvector confirmed; graph store trigger-gated, not roadmapped**); which local models are validated for baseline quality at acceptable cost; exact 🟢/🟡 tier assignment per tool (needs a human risk review — Lars + compliance lens); egress posture (resolved — A8 revised: location ladder + secret-strip, no PII pseudonymization).
